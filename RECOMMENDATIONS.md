# LogPad 新方向建议

> 基于项目定位：真人出镜 + 本地 Agent 包装（Codex/Claude/OpenClaw 做调研/脚本/包装，人录口播素材）

---

## 一、功能层面（What to build）

### 1. 提词器模式（Teleprompter）
当前录制页面只有分段指引，没有真正的滚动提词。可以直接在浏览器里做一个全屏提词器：
- 按 Beat 分段滚动，支持速度调节
- 手机/Pad 横屏可用，当成物理提词器
- 高亮当前正在读的句子

**价值**：减少录制时的忘词和 NG 次数，提升口播效率。

**落地状态**：已完成第二阶段。录制页提供全屏提词器，支持速度、字号、暂停/继续、段落跳转和键盘控制。

### 2. Agent 封面生成器
在发布台增加一个"生成封面"功能：
- 输入标题 + 选一张视频帧或上传图片
- 通过本机 Agent dispatcher 把任务交给 Codex/Claude 路由，LogPad 不直接持有模型 API
- 提供 3 个风格变体（悬念型、数字型、对比型）

**价值**：封面是决定 CTR 的核心，目前系统完全没有涉及。

**落地状态**：已完成第二阶段。发布台可通过本机 Agent dispatcher 生成 3 个封面 brief，支持选择方案并写入发布说明；后续可接素材库或图像 agent 生成最终图片。

### 3. 素材自动关联
当前素材需要手动关联到选题。可以增加自动识别：
- 上传素材时，文件名包含选题 ID 自动关联
- 或者基于内容识别：视频/音频的元数据、图片 OCR 文字与脚本内容匹配

**价值**：减少手动操作，素材库越大越省力。

**落地状态**：已完成第一阶段。上传文件名包含选题 ID 时会自动关联到 episode，并同步写入 `runs/<episode>/05-assets/asset_manifest.json`。

### 4. 多平台一键发布（半自动）
发布台目前只有计划和文案管理。可以扩展到：
- 生成各平台要求的发布文件包（YouTube 描述、B站标签、抖音话题等）
- 导出为 `.zip` 包含：视频文件 + 文案 txt + 封面 + 标签文件
- 未来可接入各平台开放 API（B站、YouTube Data API）实现真正一键发布

**价值**：减少复制粘贴的重复劳动，降低发布出错率。

### 5. 选题灵感银行
当前选题只能手动创建。可以增加：
- 一个简单的"灵感速记"输入框（类似 Drafts），随时记录想法
- 本地 Agent 自动给灵感打标签、关联趋势关键词
- 灵感可以一键"转正"为选题

**价值**：解决"有想法时没空做、有空做时没想法"的问题。

**落地状态**：已完成桌面 + 移动端速记入口，并支持一键转成选题；Agent 自动打标签可作为下一步增强。

### 6. 创作者操作台
首页应该回答“今天先推进什么”，而不是只展示看板列：
- 汇总待补数据、临近发布、卡住超过 7 天的内容
- 给每个选题映射下一步动作
- 一键把当前状态交给 OpenClaw 做异步巡检

**价值**：减少每天打开系统后的判断成本，让 LogPad 更像工作流驾驶舱。

**落地状态**：已上线首页操作台和 OpenClaw 巡检 handoff。

### 7. 内容工厂 Workspace
`/episodes` 应该是正式工作区，而不是跳回首页：
- 搜索标题和 episode ID
- 按状态筛选
- 显示活跃选题、可录制数量、总量
- 每条内容显示 10 节点完成度、下一缺口和下一步动作

**价值**：当选题数量变多后，首页只负责“今天先做什么”，内容工厂负责“所有内容资产在哪里”。

**落地状态**：已完成。`/episodes` 已经成为独立内容工厂页面。

### 8. 开题表单
新建选题不能只收标题，需要把内容生产的第一判断固定下来：
- 标题
- 观众承诺 / 看完得到什么
- 目标平台
- 创建后去判断选题、写脚本或录制

**价值**：减少“先建了个空题，后面忘了为什么要做”的情况，并让 Codex/OpenClaw 后续能读到同一份上下文。

**落地状态**：已完成。开题信息会写入 SQLite 和 `runs/<id>/episode.json`，详情页可见。

## 二、技术层面（How to improve）

### 1. 引入 tRPC 或 Zod 统一 API 类型安全
当前 API 路由的类型和前端调用是分离的，容易出现字段改名后前后端不一致。
- **tRPC**：前后端共享 TypeScript 类型，调用即类型安全
- **Zod**：在 API 入口处做 schema 校验，替代手动的 `typeof` 检查

**优先级**：中。当前项目规模下收益有限，但随着功能增加会非常有价值。

### 2. 状态管理升级
目前所有状态都是 `useState` + `useEffect` + `fetch`，页面间数据不同步（如在 script 页面保存后回到 dashboard 看到的是旧数据）。
- 引入 **TanStack Query (React Query)**：自动缓存、后台刷新、乐观更新
- 或者更轻量的 **SWR**

**优先级**：高。这是当前用户体验的一个明显短板。

### 3. 本机 Agent Runtime 接入
生成能力不放在 LogPad 产品层。LogPad 只发结构化任务，本机 runtime 决定用 Codex、Claude 路由或 OpenClaw handoff：
- **Codex**：脚本、标题、Hook、复盘、Remotion 需求、工作流写回
- **Claude CLI/CCS 路由**：需要不同模型风格时作为 Agent 内部 provider
- **OpenClaw**：提醒、巡检、发布 checklist、异步 handoff
- **本地 Python/Remotion 脚本**：props 生成、QC、发布包等确定性执行

**优先级**：高。这和当前本机多 Agent 操作系统一致，也避免产品内硬编码模型和 key。


### 4. 视频处理管道
当前没有视频处理功能。可以考虑集成：
- **FFmpeg.wasm**：浏览器端视频压缩、格式转换、截取封面帧
- 或后端调用系统 FFmpeg：自动生成视频缩略图、提取音频波形图、检测视频时长

**优先级**：中。视频缩略图目前是前端 canvas 截取，不够稳定。

### 5. 数据库迁移到更灵活的方案
目前 SQLite 足够用，但如果未来需要：
- 多端同步（手机、Pad、电脑）
- 协作（多人团队）
- 云备份

可以考虑：
- **Drizzle ORM + Turso**（SQLite 的分布式版本，边缘部署）
- 或保持 SQLite 但增加 **Litestream** 实时备份到 S3

**优先级**：低。当前单机方案完全够用。

## 三、概念层面（How to think）

### 1. 从"工具"到"工作流引擎"
目前 LogPad 是一个功能集合（看板、编辑器、发布台）。可以进一步抽象为**可编排的工作流**：
- 定义每个选题的"标准工作流模板"
- 模板包含：必须的 stage、每个 stage 的 checklist、deadline 提醒
- 新选题创建时自动套用模板

**类比**：Notion 模板 vs 空页面；Linear 的 workflow vs 普通看板。

### 2. 数据驱动的选题决策
当前评分系统（curiosity/audience/platform/feasibility）是静态的。可以引入动态数据：
- 记录每个选题从 idea → 发布 的完整周期时长
- 分析哪类选题的 ROI 最高（投入时间 vs 播放量）
- 用这些数据反哺选题阶段的 Agent 建议

**核心概念**：创作者也需要自己的"数据飞轮"。

### 3. 内容资产化管理
把每期内容视为"资产"而非"一次性产品"：
- 建立"素材复用库"：好的 Hook、转场、BGM、封面模板可跨选题复用
- 内容版本管理：抖音 60s 版、B站 5min 版、YouTube 8min 版，都是同一选题的不同"衍生品"
- 年度/季度内容审计：哪些选题系列表现最好，值得继续做第二季

**核心概念**：自媒体不是项目制，是资产管理制。

### 4. "人机回环"（Human-in-the-loop）设计
当前生成能力是"一键生成"，但真人出镜内容的灵魂在于人的判断。更好的设计是：
- Agent 生成 3 个选项 → 人选择/编辑 → Agent 根据反馈优化
- 每次使用 Agent 后弹出"这个结果有用吗？"的反馈按钮
- 用反馈数据微调本地 prompt 模板

**核心概念**：Agent 是副驾驶，不是自动驾驶。

## 四、工具层面（What to use）

### 立即可以引入的工具

| 工具 | 用途 | 成本 | 优先级 |
|------|------|------|--------|
| **TanStack Query** | 替代手写 fetch + useEffect，自动缓存和刷新 | 免费 | 高 |
| **Zod** | API 参数校验 + 类型推导 | 免费 | 高 |
| **Vitest** | 单元测试（比 Jest 快，Vite 原生支持） | 免费 | 高 |
| **本机 Agent dispatcher** | LogPad 发任务，Codex/Claude/OpenClaw 执行 | 已有 | 高 |
| **Playwright** | E2E 测试关键用户流程 | 免费 | 中 |
| **shadcn/ui** | 替换部分手写组件（Dialog、Select、Calendar），提升一致性和无障碍 | 免费 | 中 |
| **date-fns** | 替代手写的日期格式化逻辑（distribution 页面有很多日期计算） | 免费 | 低 |
| **clsx + tailwind-merge** | 替代内联的 `cn()` 函数，更 robust | 免费 | 低 |

### 未来可考虑的工具

| 工具 | 用途 | 成本 |
|------|------|------|
| **OpenClaw gateway** | 异步巡检、提醒、发布 checklist 执行 | 已有 |
| **FFmpeg.wasm** | 浏览器端视频处理 | 免费 |
| **Supabase** | 如果未来需要云端数据库 + 认证 | 有免费额度 |
| **Plausible** | 轻量自托管分析，了解 LogPad 自己的使用数据 | 免费自托管 |
| **n8n** | 工作流自动化，比如"发布成功后自动发一条 tweet" | 免费自托管 |

## 五、路线图建议

### Phase 1 — 夯实基础（1-2 周）
1. 修复 Review 中的 HIGH/MEDIUM 级别问题（已完成）
2. 引入 TanStack Query + Zod，重构核心 API 调用（已完成基础设施）
3. 补充基础测试（已完成 21 个单元测试）

### Phase 2 — 体验升级（2-3 周）
1. 提词器模式（已完成第一阶段）
2. Agent 封面生成器（已完成 brief 生成）
3. 素材自动关联（基于文件名规则，已完成第一阶段）
4. 创作者操作台（已完成第一阶段）
5. 内容工厂与开题表单（已完成第一阶段）

### Phase 3 — 智能化（3-4 周）
1. 本机 Agent routing 深化（Codex/Claude/OpenClaw 分工、任务状态、写回；已完成 handoff 入口）
2. 数据飞轮：选题 → 发布 → 数据 → 选题建议
3. 内容资产库（复用 Hook、封面模板）

### Phase 4 — 生态扩展（按需）
1. 多平台一键发布 API 集成
2. 移动端 PWA 体验优化（目前侧栏在手机上可用，但可以更好）
3. 协作支持（多账号、审稿流程）

## 结语

LogPad 已经具备了一个"创作者操作系统"的雏形。接下来的关键不是加更多功能，而是：
1. **让现有功能更可靠**（测试、类型安全、性能）
2. **让 Agent 更深地嵌入工作流**（不只是生成文案，而是参与每个决策点）
3. **让数据形成闭环**（从直觉创作转向数据辅助创作）

你现在的方向（真人出镜 + 本机 Agent 包装）非常适合这个系统的演进路径——人在镜头前提供真实感和信任感，Codex/OpenClaw/Claude 路由在背后处理可自动化的环节。LogPad 就是这个"人机协作"的中枢。
